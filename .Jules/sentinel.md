## 2024-05-19 - [HIGH] React Markdown XSS 취약점 수정 **Vulnerability:** ReactMarkdown rendered `a` tags without sanitizing `href` attributes for `javascript:`, `vbscript:`, or `data:` schemes, posing a Cross-Site Scripting (XSS) risk across `KnowledgePanel.tsx`, `ChatPanel.tsx`, and `console/page.tsx`. **Learning:** `react-markdown` escapes HTML by default, but URL schemes like `javascript:` inside markdown links (e.g., `[click](javascript:alert(1))`) were passed directly to custom `a` component renderers. **Prevention:** Implement an `isSafeUrl` check within `markdownComponents` to strictly deny execution-oriented protocols and sanitize `href` before assigning it to an anchor or click handler.

## 2024-05-24 - [HIGH] Missing Rate Limiting on Write/Mutate API Endpoints **Vulnerability:** The API endpoints for creating follows (`POST /api/follow`), creating expeditions (`POST /api/expedition`), and updating expeditions (`PATCH /api/expedition/[id]`) lacked rate limiting. This could allow malicious users to spam requests, leading to database exhaustion (DoS) or abuse of the application features. **Learning:** While rate limiting was correctly applied to heavy read operations (like AI generation in `/api/topic`), state-mutating API routes were overlooked, assuming low natural request volume. **Prevention:** Always apply `checkRateLimit` from `@/lib/rate-limit` to any `POST`, `PATCH`, or `PUT` endpoints that perform database mutations, with an appropriate window (e.g., 1000ms - 3000ms).
## 2024-05-25 - [HIGH] Missing Rate Limiting on Unauthenticated Registration API **Vulnerability:** The `/api/auth/register` endpoint lacked rate limiting, making it vulnerable to denial of service (DoS) and account creation spam since any user could repeatedly call the endpoint. **Learning:** While rate limiting was applied to authenticated routes using `userId`, unauthenticated routes require identifying the user by IP address using `x-forwarded-for` or `x-real-ip` headers. **Prevention:** Always apply `checkRateLimit` to unauthenticated endpoints using IP addresses, and ensure to mock these headers in tests to prevent false failures.
