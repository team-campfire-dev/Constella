## 2024-05-19 - [HIGH] React Markdown XSS 취약점 수정 **Vulnerability:** ReactMarkdown rendered `a` tags without sanitizing `href` attributes for `javascript:`, `vbscript:`, or `data:` schemes, posing a Cross-Site Scripting (XSS) risk across `KnowledgePanel.tsx`, `ChatPanel.tsx`, and `console/page.tsx`. **Learning:** `react-markdown` escapes HTML by default, but URL schemes like `javascript:` inside markdown links (e.g., `[click](javascript:alert(1))`) were passed directly to custom `a` component renderers. **Prevention:** Implement an `isSafeUrl` check within `markdownComponents` to strictly deny execution-oriented protocols and sanitize `href` before assigning it to an anchor or click handler.

## 2024-05-24 - [HIGH] Missing Rate Limiting on Write/Mutate API Endpoints **Vulnerability:** The API endpoints for creating follows (`POST /api/follow`), creating expeditions (`POST /api/expedition`), and updating expeditions (`PATCH /api/expedition/[id]`) lacked rate limiting. This could allow malicious users to spam requests, leading to database exhaustion (DoS) or abuse of the application features. **Learning:** While rate limiting was correctly applied to heavy read operations (like AI generation in `/api/topic`), state-mutating API routes were overlooked, assuming low natural request volume. **Prevention:** Always apply `checkRateLimit` from `@/lib/rate-limit` to any `POST`, `PATCH`, or `PUT` endpoints that perform database mutations, with an appropriate window (e.g., 1000ms - 3000ms).## 2024-05-30 - [Missing Rate Limiting on Unauthenticated Endpoints] **Vulnerability:** The `POST /api/auth/register` and `DELETE /api/expedition/[id]` routes lacked rate limiting, allowing unauthenticated or malicious users to perform DoS attacks or spam database records. **Learning:** Unauthenticated endpoints like user registration are highly susceptible to bot attacks, requiring IP-based rate limiting as a defense mechanism when user IDs are not available. **Prevention:** Always assess unauthenticated endpoints and sensitive destructive operations (like DELETE) for rate limit coverage using existing utilities like `checkRateLimit`.
## 2025-03-30 - [ReactMarkdown External Link XSS] **Vulnerability:** [External links in ReactMarkdown could execute XSS payloads via javascript: or data: URIs if not properly sanitized. The legacy inline logic was duplicated and brittle.] **Learning:** [Client-side URL sanitization in Markdown renderers is critical to prevent malicious scripts from executing when a user clicks a link.] **Prevention:** [Centralize URL validation in a single utility using the URL constructor and an allowlist of safe protocols (`http:`, `https:`, `mailto:`, `tel:`), and fall back to strict prefix checks for relative URLs.]

## 2025-01-21 - [Rate Limiter Memory Leak] **Vulnerability:** [The in-memory `rateLimitMaps` in `src/lib/rate-limit.ts` grew indefinitely without a cleanup mechanism, risking a Denial of Service (DoS) through memory exhaustion.] **Learning:** [In a long-running Node.js process (e.g., handling SSE streams), keeping map references for every user/IP request over time causes unbounded memory growth, especially against unauthenticated routes.] **Prevention:** [Implement a periodic `setInterval` to iterate and clean up expired Map entries and empty Maps. Ensure `.unref?.()` is called on the interval to prevent blocking graceful server shutdowns.]

## 2025-05-24 - [HIGH] Missing User Validation Before Content DB Upsert **Vulnerability:** Unauthenticated or malicious users could create "ghost users" in the Content DB by providing arbitrary `recipientId` or `targetUserId` values to `/api/dm`, `/api/follow`, or `/api/expedition/[id]/members` endpoints. These IDs were upserted directly into the Content DB without verifying their existence in the Main DB. **Learning:** While the primary Main DB enforces constraints on users, the Content DB (which operates separately for scalability/architecture reasons) was blindly trusting client-provided IDs. Failure to cross-validate allows attackers to pollute the secondary database and potentially bypass access controls via Insecure Direct Object References (IDOR). **Prevention:** When handling requests that involve user IDs provided by clients, always verify the user exists in the Main DB (`prisma.user.findUnique`) before upserting or writing them to the Content DB (`prismaContent.user`).
